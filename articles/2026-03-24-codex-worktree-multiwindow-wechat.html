我第一次用 Codex 的 worktree 多窗口写代码，真正麻烦的不是命令

前天我还在研究，怎么让 Codex 用多分支、多窗口的方式写代码，也就是 worktree 这套方法。那时我主要在看技术文档，里面有不少命令行细节。今天我真正上手做了一次，感觉很明确：命令不算最难，真正麻烦的，其实是协作。

这次我没有完全照着那些繁琐步骤去做，而是先开了一个主控 Codex 窗口，把需求告诉它，让它帮我做规划、执行一部分 Git 和 worktree 操作。我按照主控窗口的指引，先生成目录结构，再生成两个分支的目录文档，还让它把两个分支窗口最初要用的提示词也准备好。

我这次要做的是一个视觉错觉游戏网站。这个项目刚好适合用 worktree 来拆，因为它天然能分成两部分。

一部分是内容，也就是题目、文案、节奏、分享文案这些，可以放到 content-lab。

另一部分是玩法和页面实现，也就是首页、答题页、结果页、状态逻辑、评分、部署这些，可以放到 game-engine。

主控窗口则负责更上层的事情，比如审查、汇总、合并和收口。项目里的 docs/worktrees.md 后来也把这个边界固定下来了。

接着，我打开两个分支窗口运行 Codex，同时也让主控窗口替我执行对应的 Git 命令和 worktree 命令。这样三个窗口就同时开始工作了。

这样做的好处很快就能感受到。最明显的一点，就是上下文压力确实比单窗口小一些。因为每个窗口只盯着自己那部分事情，窗口上下文更窄，输出也更稳一些。

但问题也马上出来了。

第一个问题是，主控窗口并不能直接看到另外两个窗口到底在做什么。它可以读取整个目录的文件变化，所以能检查、能审计，但它看不到另外两个窗口里的对话过程。于是等两个分支窗口完成工作以后，还是需要把各自的情况汇总给主控窗口，否则主控窗口只能靠读文件变化来反推发生了什么。这很可能会增加额外的 token 消耗。

第二个问题更实际，就是窗口辨识很麻烦。现在这些窗口没有特别清晰的名字，看起来都像普通的 cmd 终端。你必须非常清楚哪个窗口对应哪个 worktree。如果判断错了，把提示词或者命令发错地方，就会出问题。我第一次就把提示词复制错了窗口，导致工作暂停。还好没有造成坏结果，但已经足够说明风险。

更不方便的是，当前窗口到底在哪个目录，很多时候还得看 Codex 窗口最底部的路径显示，并不顺手。

还有一个问题，是进度不同步。两个窗口并行，不代表它们会同时完成。只要一边快、一边慢，主控窗口后面的汇总、合并和下一步动作，还是得等两边都交代清楚后才能推进。所以在我这次这种比较初级的 worktree 实战里，多窗口更像是把工作切开、把上下文压窄，而不是消灭等待。

最后，这次项目还是做完了。前面分支各自推进，后面回到主控窗口，通过自然语言让它帮我完成检查、合并、完善、本地部署和显示这些事情。应该说，这还是一次比较简单、比较初级的 worktree 开发。

不过它已经让我看清楚一件事：以后稍大一点的项目，多窗口协作大概率是必需的。否则上下文一长，Codex 一旦开始“失忆”，代码质量就很难稳定。

也正因为这样，我开始想到 Codex App。它已经有 Windows 版本了。据说如果用这个 App，主控窗口可能更方便地监控各个分支窗口的状况。如果这点成立，那它对多窗口开发会很有帮助，因为现在最大的成本，不是不会开 worktree，而是主控窗口对其他窗口的过程感知还不够强。

所以我这次真正得到的结论不是“我学会了几条命令”，而是：worktree 这件事，重点从来不只是 Git，而是你有没有能力把分工、边界、回报和合并这几件事讲清楚。
