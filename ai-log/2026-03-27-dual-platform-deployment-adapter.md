### 2️⃣ Codex 核心结论

- 问题分析：
  - 业务逻辑不需要拆成 Vercel 版和 Cloudflare 版，但部署适配层需要显式区分。
  - 当前项目更适合 `Vercel + Next.js` 与 `Cloudflare Workers + OpenNext` 组合，不适合直接走 Cloudflare Pages 静态导出。
  - Cloudflare 适配初始失败的根因有两个：
    - OpenNext 需要 Next.js `standalone` 输出。
    - 原 `outputFileTracingRoot` 让 monorepo 上级目录被卷入 standalone 结构，导致 OpenNext 在错误路径寻找 `pages-manifest.json`。
- 关键决策：
  - 保持业务代码不分叉。
  - 在 `game-engine/` 新增 Cloudflare Workers 适配文件：`open-next.config.ts`、`wrangler.jsonc`。
  - 在 `game-engine/package.json` 增加 `cf:build`、`preview`、`deploy`、`cf-typegen` 脚本。
  - 将 `game-engine/next.config.ts` 调整为 `output: "standalone"`，并移除 `outputFileTracingRoot`。
- 重要代码：
  - `game-engine/open-next.config.ts`
  - `game-engine/wrangler.jsonc`
  - `game-engine/package.json`
  - `game-engine/next.config.ts`
- 风险说明：
  - `content:sync` 不是并发安全的，同时跑多个构建会产生文件复制竞争；实际 CI/CD 顺序构建不受影响。
  - OpenNext 在 Windows 上有官方警告，建议实际 Cloudflare 发布验证优先在 Linux CI 或 Cloudflare 环境中确认。
- 最终结论：
  - 当前仓库已具备单套代码同时面向 Vercel 和 Cloudflare Workers 的部署配置。
  - Vercel 常规构建与 Cloudflare OpenNext 构建都已在本地验证通过。
